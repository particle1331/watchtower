## Proposal

| Activity | Where we are | Where we want to be | OSS option | Azure option | Tradeoff | Recommendation |
|----------|--------------|---------------------|------------|--------------|----------|----------------|
| Dataset extraction | Data is queried from databases or downloaded from Blob and cached on local disk. Refreshes are manual and happen when someone remembers. | Repeatable extraction jobs record the owner, schedule, query version, timestamp, source, row count, and checksum. | Python extraction jobs orchestrated with Prefect or Dagster. | Scheduled Azure ML pipeline or job reading from the database or Blob. | OSS is flexible but requires orchestration infrastructure; Azure ML reduces ops but creates Azure dependency and needs cloud-team provisioning. | Use scheduled Azure ML jobs for important datasets. Keep lightweight queries on demand initially. |
| Dataset versioning | Datasets are saved on disk and are not versioned, so old training data may not be reproducible. | Every run references an immutable dataset version linked to its Git commit and MLflow run. | DVC with an Azure Blob remote, or immutable Blob paths with manifest files. | Azure ML data assets pointing to immutable date- or hash-based Blob paths. | DVC is portable but adds another tool; Azure data assets integrate with Azure ML but a versioned reference is meaningless if files at the path are overwritten. | Use immutable Blob snapshots registered as Azure ML data assets. Never overwrite files at a versioned path. |
| Data validation | Validation occurs while processing data shortly before training. | Reusable schema and quality checks run when a dataset is created and before training, and failures stop the pipeline. | Pandera or Great Expectations with reports logged to MLflow. | Pandera or Great Expectations inside an Azure ML component, with reports stored in the self-hosted MLflow artifact store. | Great Expectations offers richer reports but more maintenance; Azure does not eliminate the need to define validation rules. | Start with Pandera checks in extraction and training jobs. Log reports to self-hosted MLflow 3. Adopt Great Expectations only if richer reporting becomes necessary. |
| Feature engineering | Preprocessing is rerun for training. Learned values such as means are saved and applied to production data. | One tested preprocessing pipeline is packaged with the model and used consistently in training and inference. | Framework-native pipelines such as scikit-learn Pipeline, packaged as an MLflow model. | Azure ML reusable pipeline component plus an MLflow model registered in Azure ML. | A feature store adds complexity without a current reuse requirement; packaging transformations with the model is simpler and prevents skew but couples preprocessing to the model artifact. | Package preprocessing and learned parameters with the MLflow model. Do not introduce a feature store yet. |
| Model training | CPU training runs on local machines or manually started Azure compute instances. GPU training is done manually via Azure ML compute instances, and models are downloaded by hand. | Conventional training runs locally for exploration or as a reproducible remote CPU job using a pinned environment and scale-to-zero compute; GPU and fine-tuning workloads are handled by a separate GPU compute profile. | Containerized training submitted to a VM or container worker. | Azure ML jobs on a CPU compute cluster with minimum nodes set to zero. | Local runs are fast for exploration; Azure jobs improve reproducibility but have startup time and require quota and infrastructure approval. | Keep local runs for exploration. Run candidate and conventional production training as Azure ML jobs on a scale-to-zero CPU cluster. Use a separate GPU compute profile for fine-tuning and multi-GPU experiments. |
| GPU model training / fine-tuning | GPU training and fine-tuning are done by manually creating compute instances in Azure ML. Models are trained interactively and then downloaded manually; runs are not logged to a shared registry and model lineage is implicit. | GPU training and fine-tuning run as reproducible Azure ML jobs defined in Git, log parameters, metrics, and artifacts to the self-hosted MLflow server, and register the resulting model so deployments can resolve it by exact version instead of downloading files manually. | Containerized multi-GPU training on a VM or Kubernetes cluster using PyTorch DDP, DeepSpeed, or FSDP. | Optional separate Bicep module that attaches a GPU compute cluster to the existing Azure ML workspace; job definitions remain in Git. | Raw VMs are flexible but require manual driver, NCCL, and scaling setup; Azure ML provides quota-managed, reproducible GPU jobs with environment and data integration. | Deploy the optional GPU compute module once GPU quota is approved. Run a multi-GPU fine-tuning POC as a Git-defined job that logs to self-hosted MLflow and registers the model. Stop manual downloads for production artifacts. |
| Compute provisioning | Compute is started manually. A separate cloud team owns Azure infrastructure, so changes can be delayed. | Approved platform and compute profiles are provisioned through reusable infrastructure-as-code, allowing the ML team to operate MLflow and submit jobs without creating resources manually. | Terraform modules for container workers and self-hosted MLflow. | Bicep or Terraform module for the MLflow Container App, PostgreSQL, Blob Storage, Key Vault, identities, networking, monitoring, Azure ML workspace, and a scale-to-zero CPU compute cluster. GPU compute is deployed through a separate optional module once quota is approved. | Infrastructure-as-code removes repeated tickets but self-hosted MLflow adds a control plane that the team must patch, back up, restore, and monitor. | Request one reusable platform module covering self-hosted MLflow 3, a scale-to-zero CPU compute profile, and an Azure ML workspace. Keep GPU compute in a separate optional module to avoid blocking core platform deployment on GPU quota. |
| Experiment tracking | Git, configuration files, and Excel sheets provide partial tracking. MLflow is not currently used. | Every run records parameters, metrics, artifacts, environment, Git SHA, dataset version, prompt version, evaluation result, and LLM traces in MLflow 3. | MLflow 3 on Container Apps, PostgreSQL backend, and proxied Azure Blob artifact storage. | Azure ML managed MLflow is limited to older MLflow APIs; Azure Databricks offers managed MLflow 3 at higher platform cost. | Self-hosting provides current MLflow tracking, evaluation, prompt, and tracing APIs but makes image upgrades, schema migrations, authentication, backup, restore, and monitoring the team's responsibility. | Self-host a pinned MLflow 3 image on Container Apps with PostgreSQL and proxied Blob artifacts. Assign an owner and test backup and upgrade procedures before onboarding projects. |
| Model registry | Artifacts are packaged and copied manually, and deployment history is implicit. | Approved models are immutable, versioned, linked to their originating run, and promoted through a recorded approval. | MLflow 3 Model Registry with aliases such as candidate staging and champion. | Azure ML model versions and tags, or Databricks Unity Catalog. | Aliases are convenient mutable pointers but do not enforce approval themselves; one shared registry also increases the blast radius of accidental changes. | Register only models that pass evaluation. Allow only CI to move protected aliases after GitHub Environment approval and deploy the exact resolved model version recorded in the release manifest. |
| Model evaluation | Backtesting uses normal dataset splits. Smoke examples live in Excel, and review decisions are made in Zoom. | Evaluation is executable, versioned, and repeatable. Thresholds block regressions and qualitative approval is recorded. | pytest and custom evaluation code with results logged to MLflow. | Azure ML evaluation job triggered by GitHub Actions, with reports logged to managed MLflow. | Automation requires converting Excel cases into machine-readable data; human review remains necessary but the decision and evidence become auditable. | Run a small suite on pull requests, a full suite before production, and record a human approval. |
| Prompt development | Prompts are embedded in code and committed to Git without explicit prompt versions. | Prompts are reviewed in Git, registered as immutable MLflow prompt versions, and promoted by alias; each run records the prompt version, Git SHA, model deployment, and parameters. | MLflow 3 Prompt Registry and GenAI tracing. | Microsoft Foundry prompt tooling or Git-only prompt files. | Maintaining prompt source in Git and deployable versions in MLflow adds a synchronization rule, but it gives code review plus runtime promotion and rollback. Avoid Prompt Flow because its retirement is scheduled for April 2027. | On merge register the reviewed prompt as an immutable MLflow version. Allow CI to move candidate staging and champion aliases after evaluation and approval; deployments record the exact resolved version. |
| Prompt evaluation | A few smoke examples are tracked in Excel. Evaluation is mostly manual and is not tied to a Git commit. | A versioned golden dataset covers expected behavior, edge cases, safety cases, and regression thresholds. Every prompt change is evaluated and linked to its registry version. | MLflow 3 GenAI evaluation or custom evaluators with pytest. | Microsoft Foundry evaluation or a custom Azure ML job calling Azure OpenAI. | LLM evaluation consumes tokens and judge metrics can vary; deterministic checks are cheaper and more stable. | Start with deterministic checks and human-labelled examples in MLflow 3 before adding LLM-as-judge evaluation. |
| Batch inference | For the 30000-doc/day LLM workload the team runs a compute instance 24/7 that polls Azure Blob for the previous day's uploads, processes them, then sits idle and billing until the next day's poll. There is no queue and no scaling; the compute is paid for continuously regardless of whether documents are present. | Portable task orchestration: self-hosted Redis on Container Apps feeds Celery workers as both broker and result backend. Workers scale on queue depth and then to zero. Releases use immutable images and versioned Redis list keys so stale revisions cannot fetch new task contracts. Processing is idempotent, rate-limit aware, observable, and crash-safe with a Celery Beat reconciliation producer that recovers queue state lost to Redis restarts. | Celery with self-hosted Redis on Container Apps as broker and result backend (7-day expiry), versioned list keys, and idempotency-keyed tasks. Reconciliation via Celery Beat schedule, no separate service required. | Celery workers on Azure Container Apps with KEDA Redis list scaling; ACR stores immutable images and Application Insights receives structured telemetry. | Celery provides retries, acknowledgements, routing, and workflow primitives. Self-hosted Redis keeps the stack Azure-native at ~$6/month versus a managed broker but requires the team to monitor and restart it. Redis at-least-once delivery is safe because every task carries an idempotency key and the Beat reconciliation producer repairs gaps automatically. Versioned list keys add release discipline with temporary key overlap. | Use self-hosted Redis on Container Apps as Celery broker and result backend with 7-day result and idempotency-key expiry. Each release consumes only its versioned Redis list key. Deploy and smoke-test the new immutable revision, switch producers to the new key, drain the old key until LLEN reports zero, then deactivate the old revision. Every task carries a contract version, release, idempotency key, and exact model and prompt versions. |
| Online inference | Some projects predict on a few dozen examples every five minutes using an Azure Function sidecar. | The smallest deployment mechanism satisfying latency and reliability requirements serves a stable API with an exact immutable model or prompt reference. | MLflow model server in a container. | Azure Functions for low volume or Azure ML managed online endpoints for dedicated model serving. | Functions are inexpensive and familiar but require custom model loading; managed endpoints provide rollout and monitoring but always-on compute may exceed budget. | Retain Azure Functions for low-volume workloads. Resolve a registry alias during release and deploy the exact MLflow model or prompt version; do not fetch a mutable alias on every request. |
| Packaging | A model artifact is manually copied and placed beside an Azure Function. | One immutable package contains the model, preprocessing, dependencies, and version metadata. | MLflow Models plus Docker. | Application containers in ACR identified by immutable digest. | MLflow works well for standard models; custom containers remain necessary for application and LLM logic. | Store conventional models in self-hosted MLflow 3 and package application logic as immutable ACR containers. Include preprocessing in the model and exact artifact versions in the release manifest. |
| Deployment | Azure Functions are deployed manually through VS Code. | GitHub Actions deploys an approved artifact to test, runs smoke tests, requests production approval, and records the deployed version. | GitHub Actions deploying containers to any supported platform. | GitHub Actions with workload identity deploying to Azure Functions, Container Apps, or Azure ML endpoints. | Automation requires identities and permissions from the cloud team; manual deployment is faster to start but error-prone and unaudited. | Replace VS Code deployment with GitHub Actions using workload identity. Deploy to test, run smoke tests, then require GitHub approval for production. |
| Rollback | A team member manually performs rollback. | A workflow redeploys the previous immutable image, model, prompt, configuration, producer route, and worker queue in less than 15 minutes. | Registry alias reassignment and redeployment of the prior container digest. | Container Apps revision activation, immutable Function packages, and GitHub workflows. | A stale worker rollback is safe only when task contracts are compatible; reactivating an old revision against a new queue can corrupt or reject tasks. | Record each release's image digest, exact model and prompt versions, task contract, queue, and configuration. Roll back producer routing and revision together, and require a compatibility check before replaying tasks across queue versions. |
| CI/CD | GitHub Actions and ACR are available, but ML lifecycle automation is limited. | Pull requests run unit, data-contract, smoke, and evaluation tests. Merges build immutable images and register immutable artifacts; production alias and route changes require approval. | GitHub Actions plus MLflow 3 APIs and AMQP administration APIs. | GitHub Actions, ACR, Azure workload identity, Azure ML CLI or SDK, and GitHub Environments. | Full evaluations can be slow and incur LLM costs; queue draining also makes worker deployment asynchronous rather than a single image update. | Extend GitHub Actions with separate fast PR tests and full release evaluation. Treat registry alias changes and producer queue switches as auditable deployment steps. |
| Environments | Test and production exist. Staging support varies by service. | Gitflow governs code integration; one shared MLflow registry holds immutable artifacts; protected aliases and release manifests govern promotion into isolated test and production deployments. | Separate registries or one registry with aliases and RBAC. | Separate resource groups for runtime services plus GitHub Environments for approval. | One registry avoids copying artifacts but mutable aliases can be changed accidentally. Git branches are not environment isolation and must not be used as deployable artifact references. | Use one backed-up MLflow registry with candidate staging and champion aliases. Only CI may move protected aliases after GitHub approval; test and production deploy exact resolved versions into separate resource groups and identities. |
| Service monitoring | Logs are collected and archived to Blob, but operational and cost monitoring is limited. | Azure Monitor dashboards and alerts cover MLflow, Redis, and PostgreSQL health; failure rates; latency percentiles; queue depth; retries; deployed versions; and Azure OpenAI token usage. Log Analytics provides custom KQL workbooks for per-prediction tracing and contract-version drilldown. Application Insights captures structured telemetry with sampling. | OpenTelemetry exported to a self-hosted or managed observability stack. | Application Insights, Azure Monitor, Log Analytics, and Azure Cost Management. | Azure monitoring is deeply integrated; ingestion can grow, and an ingestion cap that cuts data collection makes the team blind. Cost Management budgets do not stop spending. | Use Azure Monitor for real-time service health with KQL dashboards, Log Analytics for custom investigation, and Application Insights for structured telemetry with cost-conscious sampling. Use Cost Management budgets and forecast alerts for Azure spend; use the Log Analytics ingestion cap only as an emergency safeguard, not a spending cap. |
| Quality monitoring | Prediction quality is manually reviewed every few weeks and takes about one day. User feedback is collected in a Teams channel. | Production samples and feedback feed a versioned evaluation dataset, and a scheduled job produces quality reports and regression alerts. | Evidently or MLflow 3 evaluation and monitoring jobs. | Scheduled Azure ML evaluation job, with structured Teams feedback exported to Blob or Table storage and linked to prediction IDs. | Automated proxies cannot fully replace labels or human review; collecting production samples has storage and privacy implications. | Automate data collection and MLflow 3 report generation first to reduce the recurring review to under two hours. Link Teams feedback to prediction IDs. |
| Audit and governance | Deployment decisions occur in Zoom, while application logs are saved to Blob. | Each release records its approver, evaluation run, dataset, code SHA, image digest, exact model and prompt versions, task contract, queue route, and deployment time. | MLflow tags and aliases plus GitHub releases and approval records. | GitHub Environment approvals, Azure activity logs, and Blob retention. | A heavy governance platform is unnecessary for the current team but mutable aliases and queue routes require an audit trail. | Record approvals in GitHub rather than Zoom alone. Each release links to its MLflow run and evaluation, exact artifact versions, Git SHA, dataset, image digest, task queue, contract version, and approver. |

## Risks

| Area | Hard question | Why it matters | Assumption at risk | Mitigation |
|------|---------------|----------------|--------------------|------------|
| Cloud-team dependency | Can the cloud team provision the MLflow Container App, PostgreSQL, Blob, Key Vault, Entra integration, networking, Azure ML compute, and worker environment inside two weeks? | Tracking, registry, training, and deployment all depend on approved infrastructure. | Assumption: the complete platform request can be delivered in phase 0. | Submit one complete IaC and RBAC request on day 1. Do not start the ten-week clock until an ETA is accepted; instrument locally without creating a temporary server if provisioning slips. |
| Self-hosted ownership | Who owns MLflow availability, vulnerabilities, upgrades, database migrations, backups, and incident response? | Self-hosting removes the managed-service boundary and creates permanent operational work. | Assumption: a five-person team can absorb a control-plane service. | Name a primary owner and deputy, budget recurring maintenance time, pin server and client versions, and reconsider managed MLflow 3 if ownership becomes unsustainable. |
| MLflow upgrade safety | Can the team upgrade MLflow without corrupting registry and trace metadata? | MLflow database migrations may be nontransactional and require server coordination. | Assumption: normal Container Apps revision rollout is sufficient for MLflow upgrades. | Stop writes, take a PostgreSQL backup, test the migration in a restored test database, run one migration job, deploy the matching pinned server, verify, and document rollback. |
| MLflow recovery | Have PostgreSQL point-in-time restore and Blob artifact recovery been tested together? | A database-only restore can leave metadata pointing at missing artifacts, while Blob-only recovery cannot reconstruct registry state. | Assumption: managed backups imply a usable MLflow recovery procedure. | Enable PostgreSQL PITR and Blob soft delete/versioning, define RPO/RTO, run a restore exercise in phase 0 and quarterly, and protect resources from accidental deletion. |
| MLflow authentication | Will Entra authentication work for browsers, local MLflow clients, GitHub Actions, and Azure ML jobs without shared long-lived credentials? | An authentication design that only supports the UI will block automation; a broad shared token removes accountability. | Assumption: Container Apps or the selected MLflow auth integration supports every caller. | Prototype all four caller types before onboarding projects. Use workload identities and app roles where possible, keep unavoidable secrets in Key Vault, and deny unauthenticated ingress. |
| MLflow availability | Is one MLflow replica acceptable and what happens to training or CI during an outage? | A single control-plane replica can restart and temporarily block writes even though deployed inference continues. | Assumption: control-plane downtime is tolerable for this team. | Set an availability objective, health alert, client retry policy, and incident runbook. Start with one replica for operational simplicity; add tested multi-replica operation only if measurements justify it. |
| Registry promotion | Can mutable candidate, staging, and champion aliases be changed only through approved automation? | A manual alias change can deploy an unevaluated model or prompt even when Gitflow branches are correct. | Assumption: team discipline alone protects production aliases. | Restrict alias mutation to CI credentials, require GitHub Environment approval, record old and new exact versions, and reconcile deployed manifests against registry state. |
| Environment isolation | Does one shared registry create an unacceptable production blast radius? | Test and production share model and prompt metadata even when runtime resource groups and identities are separate. | Assumption: immutable versions, RBAC, approvals, and backups provide enough isolation for five users. | Keep runtime resources and identities separate, deploy exact versions rather than aliases, monitor destructive operations, and split registries later if policy requires stronger isolation. |
| Budget realism | Does the budget include always-on MLflow, PostgreSQL, Redis, monitoring, and 30000-document Azure OpenAI usage? | The LLM cost and always-on control-plane services can exceed a lean infrastructure estimate. | Assumption: the stated infrastructure budget covers four always-on container replicas (MLflow, Redis) and burstable PostgreSQL. | Price every fixed and variable component before approval. Track Azure with Cost Management budgets and alerts; do not describe log ingestion caps as spending caps. The approximate Azure baseline is $110–120/month excluding OpenAI. |
| Team capacity | Can two engineers and the platform owner commit enough time for ten weeks while still shipping products? | Self-hosted MLflow and Redis add security, recovery, and operations work beyond application code. | Assumption: explicit weekly allocation is available. | Approve per-person weekly hours before starting. If capacity is lower, deliver MLflow and conventional ML first and defer the production batch feedback loop. |
| Redis broker ownership | Who monitors Redis, handles restarts, and recovers lost queue state? | Self-hosted Redis has no SLA — a restart without persistence loses queued but unacknowledged messages. In-flight tasks survive, but queued tasks vanish. | Assumption: Container Apps restart is fast enough and the reconciliation producer handles any gap. | Configure Redis with enough memory, set maxmemory-policy to noeviction, monitor via Azure Monitor, test restart recovery end-to-end, and verify the Celery Beat reconciliation producer republishes lost tasks idempotently within the agreed scan interval. |
| Stale worker code | How is an old Container Apps revision prevented from fetching a task produced for new code? | Queue workers ignore HTTP traffic weights; if revisions share a Redis list key, stale code can accept a new task contract during rollout or rollback. | Assumption: an image rollout automatically drains old key consumers before new work arrives. | Use release-specific Redis list keys and multiple-revision mode. Smoke-test the new revision on its key, switch the producer route, drain the old key until LLEN reports zero, then deactivate it. Include and validate a task contract version in every message. |
| Worker termination | What happens when Container Apps scales in or deactivates a revision during a long LLM task? | Container shutdown can interrupt a task; early acknowledgement can lose it and redelivery can duplicate side effects. | Assumption: Celery graceful shutdown always finishes before the platform terminates the container. | Use warm SIGTERM handling, late acknowledgement, reject-on-worker-loss, low prefetch, bounded task duration, external checkpoints, and idempotent writes. Test normal scale-in and forced termination. |
| Queue rollback compatibility | Can old code safely process tasks already published with a newer contract? | Reactivating the prior image and pointing it at the new key can turn an application rollback into data corruption. | Assumption: container rollback and key rollback are the same operation. | Record contract compatibility in the release manifest. Route producers and revisions together; incompatible tasks are logged and skipped by the old worker unless explicitly transformed and replayed. |
| Batch capacity | Does Redis and KEDA scaling respect Azure OpenAI RPM and TPM limits and the daily completion window? | More workers can increase 429 responses without increasing throughput. Redis has no built-in rate limiting. | Assumption: queue-depth autoscaling is sufficient capacity control. | Calculate capacity from p95 tokens per document and the processing SLA. Cap replicas and Celery concurrency at the approved quota, honor rate-limit headers, and alert on LLEN growth and sustained throttling. |
| Batch reconciliation | How are tasks lost during a Redis restart recovered? | Redis without persistence loses queued-but-unacknowledged messages; Celery cannot retry what it never dequeued. In-flight tasks survive; the Beat reconciliation producer must recover the rest. | Assumption: the reconciliation schedule and idempotency keys together close the gap reliably. | Publish transactionally from the uploader when possible. Run a Celery Beat reconciliation schedule, diff unprocessed documents against Redis result keys, and republish missing work with the same idempotency key. Test Redis restart recovery end-to-end. |
| Task state and idempotency | Where is authoritative per-document state stored and how are duplicate deliveries handled? | Redis and Celery provide at-least-once behavior, not exactly-once side effects. Redis result expiry means state is temporary. | Assumption: Redis result backend with 7-day TTL is sufficient; long-term history lives in MLflow evaluation artifacts and structured logs. | Use a Redis key per idempotency key with a 7-day TTL; atomic SETNX prevents duplicate side effects. Store exact release metadata in the result and in structured logs. Keep long-term quality history in MLflow, not in the broker. |
| Evaluation maintenance | Who maintains the golden dataset as failures and feedback accumulate? | An unmaintained suite becomes a rubber stamp. | Assumption: converting Excel cases once creates a durable quality process. | Assign a rotating owner, review monthly, add production failures after privacy review, and log the exact dataset version with every evaluation. |
| LLM model changes | Which Azure OpenAI model version and upgrade policy is deployed and how are announced upgrades evaluated? | Model upgrades, content-policy behavior, and distribution shift can alter outputs without an application release. | Assumption: weekly evaluation catches regressions quickly enough. | Record deployment and model versions, select an explicit upgrade policy, run a daily mini-evaluation and a pre-upgrade suite, and alert on real-time quality proxies. |
| Data privacy | Can prompts, traces, production samples, task payloads, and feedback contain PII or sensitive business data? | MLflow artifacts, Redis, logs, and evaluation stores may widen access or retention. | Assumption: all captured inputs and outputs may be stored unchanged. | Classify and redact data before storage, keep payloads out of broker messages where possible, log metadata rather than full text by default, restrict access, and define retention and deletion procedures. |
| Success criteria | Who verifies rollback time, processing SLA, recovery RTO, manual review time, and platform maintenance cost? | Unmeasured criteria cannot establish whether the platform improved delivery. | Assumption: done criteria are self-evident. | Measure baselines in week 1, assign an independent verifier, capture evidence during failure and rollout drills, and compare results at the week 10 retrospective. |
| GPU quota | Can the team obtain GPU quota for the fine-tuning POC in the target region? | Without quota the multi-GPU POC cannot run and GPU training cannot be validated. | Assumption: GPU SKUs such as NC-series or A100 are available in the subscription and region. | Request GPU quota early, target a region with available capacity, and start with a single-GPU SKU before moving to multi-GPU. |
| GPU cost | Is the GPU training budget separated from the lean CPU baseline and tracked per job? | GPU VMs are an order of magnitude more expensive than CPU; an idle cluster or long-running fine-tuning job can blow the budget. | Assumption: scale-to-zero and job-level cost alerts are enough to control GPU spend. | Set minimum nodes to zero, use low-priority nodes where acceptable, set a Cost Management budget alert per GPU compute, and tear down the POC cluster after each run. |
| Multi-GPU correctness | Can the team verify that distributed training actually uses all GPUs and is not silently falling back to single-GPU or data duplication? | Incorrect distributed setup wastes money and produces invalid models. | Assumption: adding a GPU SKU to the job is sufficient for multi-GPU training. | Log GPU utilization, NCCL health, and rank info; validate a small run against a single-GPU baseline before scaling to multi-GPU. |
